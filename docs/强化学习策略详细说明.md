# 本项目强化学习策略详细说明

## 1. 一句话结论

本项目里的强化学习主要负责：

```text
让 Agent 更会检索、更会选择工具、更能基于证据回答。
```

当前已经落地最完整的是：

```text
强化学习检索策略层
```

也就是把一次 RAG 检索建模成策略选择问题：

```text
用户问题
  -> 提取 query 特征
  -> 策略选择 retrieval action
  -> RAG 检索本地论文库
  -> 根据 Source Hit / Topic Hit / Point Recall 给 reward
  -> 训练策略
  -> 上线时用策略优化检索
```

所以，这里的强化学习不是直接训练 Qwen 本体，也不是直接让模型凭空生成更好的答案，而是优化 Qwen 回答前的 **检索和工具决策过程**。

## 2. 强化学习在系统中的位置

整体系统可以分成三层：

```text
第一层：Qwen / Ollama 语言模型层
  负责理解问题、规划工具调用、生成最终回答。

第二层：RAG 知识库层
  负责从本地论文库、笔记、知识块中检索证据。

第三层：强化学习策略层
  负责决定当前问题应该怎么检索、用什么 action、取多少 top_k。
```

调用路线：

```text
用户问题
  -> frontend/src/views/WorkspaceView.vue
  -> /api/chat
  -> backend/app.py
  -> backend/agent/loop.py
  -> backend/agent/tools.py::search_project_docs()
  -> backend/retrieval/policy.py::choose_retrieval_action()
  -> backend/retrieval/knowledge_store.py::search_knowledge()
  -> backend/llm/service.py::run_messages()
  -> Ollama/Qwen 回答
```

其中：

```text
RAG 负责查资料
LinUCB / DDQN / PPO / DPO / ORPO 负责学怎么查资料
Qwen 负责基于资料组织回答
```

## 3. 强化学习主要负责什么

### 3.1 当前已落地的核心任务：检索策略优化

当前项目中，强化学习最主要负责：

```text
在多个 retrieval action 中选择最合适的一个。
```

也就是说，当用户问一个问题时，系统不是固定用原始 query 去搜，而是先判断：

```text
这个问题应该普通检索？
应该偏强化学习？
应该偏轨迹规划？
应该偏论文方法？
应该偏轨迹压缩？
应该扩大 top_k？
```

这个选择过程就是强化学习策略在做的事。

### 3.2 强化学习没有做什么

需要说清楚：

```text
没有直接用 PPO/DPO/ORPO 去微调整个 Qwen 大模型。
```

当前训练对象主要是：

```text
检索策略模型 / action 选择模型
```

不是 Qwen 的全部参数。

这样设计的原因：

- 本地数据量相对有限，直接训练大模型成本高；
- 检索策略是离散动作选择，更适合小规模 RL；
- 评估指标清楚，可以用 Source Hit、Topic Hit、Point Recall 衡量；
- 对答辩来说更容易证明“RL 确实让 Agent 找资料更准”。

## 4. 强化学习的状态、动作、奖励

### 4.1 状态 State

状态来自用户 query 的特征。

代码位置：

```text
backend/retrieval/rl_env.py::features_for_query()
```

状态特征包括：

| 特征 | 含义 |
|---|---|
| `ppo` | 是否是 PPO 相关问题 |
| `dqn` | 是否是 DQN 相关问题 |
| `sac` | 是否是 SAC 相关问题 |
| 压缩/简化 | 是否涉及 trajectory compression / simplification |
| 轨迹 | 是否涉及 trajectory / 轨迹 |
| 规划/优化 | 是否涉及 planning / optimization |
| 相似度 | 是否涉及 similarity / subtrajectory / Frechet / Hausdorff / DTW |
| 奖励/偏好 | 是否涉及 reward / preference / 奖励 / 偏好 |
| 论文/实验 | 是否涉及 paper / benchmark / experiment / 论文 / 实验 |
| query 长度 | 归一化长度特征 |
| 中文字符数 | 中文问题比例 |
| 英文字符数 | 英文术语比例 |

用公式表示就是：

\[
s_t = f(q_t)
\]

其中：

- \(q_t\)：当前用户问题；
- \(f(\cdot)\)：query 特征提取函数；
- \(s_t\)：强化学习策略看到的状态。

### 4.2 动作 Action

动作是预定义的 retrieval action。

代码位置：

```text
backend/retrieval/rl_env.py::ACTIONS
```

当前动作包括：

| Action | 作用 | top_k |
|---|---|---:|
| `baseline` | 原始 query 普通检索 | 4 |
| `rl_focus` | 偏强化学习术语 | 4 |
| `trajectory_focus` | 偏轨迹领域术语 | 4 |
| `paper_focus` | 偏论文、方法、benchmark、实验 | 5 |
| `compression_focus` | 偏轨迹压缩、轨迹简化 | 5 |
| `planning_focus` | 偏路径规划、轨迹优化 | 5 |
| `similarity_focus` | 偏轨迹相似度、子轨迹搜索 | 5 |
| `reward_focus` | 偏奖励设计、逆强化学习、偏好优化 | 5 |
| `broad_search` | 扩大检索范围，拿更多证据 | 7 |

动作本质是：

```text
对原始 query 加不同的检索后缀，并调整 top_k。
```

例如：

```text
原始 query:
PPO 在 trajectory planning 中通常扮演什么角色？

如果选择 paper_focus:
PPO 在 trajectory planning 中通常扮演什么角色？
paper method benchmark ablation survey theorem experiment
```

### 4.3 奖励 Reward

奖励用来衡量这次检索动作是否真的找对了资料。

代码位置：

```text
backend/retrieval/rl_env.py::RetrievalRLEnv.step()
```

当前 reward 公式：

\[
R = 0.5 \cdot \mathrm{SourceHit}
  + 0.3 \cdot \mathrm{TopicHit}
  + 0.2 \cdot \mathrm{PointRecall}
  - 0.015 \cdot \max(\mathrm{top\_k} - 4, 0)
\]

指标含义：

| 指标 | 含义 |
|---|---|
| Source Hit | 是否命中预期论文、文件或证据来源 |
| Topic Hit | 是否覆盖预期研究主题 |
| Point Recall | 是否召回答案需要的关键点 |
| top_k penalty | 防止策略只靠扩大检索数量刷分 |

所以，本项目的 RL 优化目标不是“回答看起来更长”，而是：

```text
找对来源 + 覆盖主题 + 召回答案点。
```

## 5. 强化学习环境怎么实现

训练环境代码：

```text
backend/retrieval/rl_env.py
```

核心类：

```text
RetrievalRLEnv
```

一个 episode 的过程：

```text
1. 取一个训练问题
2. reset() 返回 query 特征 state
3. 策略选择 action_index
4. compose_retrieval_query() 生成 retrieval_query
5. search_knowledge() 调用 RAG 检索知识库
6. 根据 expected_sources / expected_topics / expected_points 计算 reward
7. 返回 state、reward、done、info
```

对应强化学习流程：

\[
s_t \xrightarrow{\;\pi_\theta\;} a_t
\xrightarrow{\;\mathrm{RAG}\;}
(r_{t+1}, \mathrm{evidence})
\]

这里的环境不是游戏地图，而是：

```text
RAG 检索系统 + 本地论文知识库 + 评测指标
```

## 6. 训练数据怎么来的

训练数据主要来自：

```text
training_data/retrieval_rl_eval_extended.jsonl
```

每条样本一般包含：

```text
query
expected_sources
expected_topics
expected_points
```

含义：

- `query`：用户问题；
- `expected_sources`：希望检索命中的论文或资料；
- `expected_topics`：希望覆盖的主题；
- `expected_points`：希望召回的答案要点。

这个数据让系统可以离线评估每个检索动作的好坏。

## 7. PPO 是怎么实现和训练的

### 7.1 PPO 在项目中的角色

PPO 主要作为：

```text
检索策略学习的强化学习对照方法。
```

它学习：

```text
在当前 query 状态下，哪个 retrieval action 的概率应该更高。
```

不是直接训练 Qwen 本体。

### 7.2 PPO 代码位置

```text
scripts/train_retrieval_policy_ppo.py
outputs/retrieval_policy_ppo_torch_60/
```

### 7.3 PPO 训练参数

主要参数：

```text
epochs = 60
rollout_size = 32
ppo_epochs = 4
minibatch_size = 16
clip_range = 0.2
value_coef = 0.5
entropy_coef = 0.02
lr = 3e-4
source_weight = 0.5
topic_weight = 0.3
point_weight = 0.2
```

PPO 的核心思想是限制策略更新幅度：

\[
L^{CLIP}(\theta)
= \mathbb{E}_t[
\min(
r_t(\theta) A_t,
\mathrm{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon) A_t
)]
\]

在本项目里：

- action 是 retrieval action；
- reward 来自 RAG 检索质量；
- advantage 来自动作相对 baseline 的收益；
- clip 防止某个动作因为偶然高分而被过度强化。

### 7.4 PPO 当前效果

当前 60 epoch PPO 结果：

| 指标 | Baseline | PPO 60 Epoch | 提升 |
|---|---:|---:|---:|
| Average Reward | 0.5171 | 0.5770 | +0.0599 |
| Source Hit | 0.3750 | 0.4769 | +10.19pp |
| Topic Hit | 0.8241 | 0.9213 | +9.72pp |
| Point Recall | 0.4120 | 0.4630 | +5.10pp |

结论：

```text
PPO 有提升，尤其提升了 Topic Hit 和 Source Hit。
但在单步检索策略任务上，综合效果不如 LinUCB。
```

答辩时可以说：

> PPO 证明了策略梯度方法可以优化检索 action，但由于当前任务更像单步 contextual bandit，所以 LinUCB 更稳定、更适合上线。PPO 更适合作为下一步多轮 Agent 工具调用策略。

## 8. DPO 是怎么实现和训练的

### 8.1 DPO 在项目中的角色

DPO 主要用于：

```text
偏好优化式的检索策略训练。
```

它不直接使用 reward model，而是从高 reward / 低 reward action 中构造偏好对：

```text
chosen action > rejected action
```

### 8.2 DPO 代码位置

```text
scripts/train_retrieval_policy_dpo.py
outputs/retrieval_policy_dpo_torch/
```

### 8.3 DPO 训练参数

主要参数：

```text
epochs = 120
batch_size = 16
lr = 1e-4
beta = 0.4
margin = 0.02
sft_coef = 0.15
source_weight = 0.5
topic_weight = 0.3
point_weight = 0.2
```

DPO 核心目标：

\[
\mathrm{preference\_logit}
= (\log p_\theta(a^+) - \log p_\theta(a^-))
- (\log p_{ref}(a^+) - \log p_{ref}(a^-))
\]

\[
L_{DPO} = -\log \sigma(\beta \cdot \mathrm{preference\_logit})
\]

项目中还加了 SFT 稳定项：

\[
L = L_{DPO} + \lambda L_{SFT}
\]

### 8.4 DPO 当前效果

当前 DPO 结果：

| 指标 | Baseline | DPO | 提升 |
|---|---:|---:|---:|
| Average Reward | 0.5028 | 0.5928 | +0.0900 |
| Source Hit | 0.3056 | 0.4444 | +13.88pp |
| Topic Hit | 0.8704 | 0.9560 | +8.56pp |
| Point Recall | 0.4444 | 0.5231 | +7.87pp |

结论：

```text
DPO 在偏好对训练下能提升 Source Hit、Topic Hit 和 Point Recall。
它适合说明“偏好优化可以让策略更倾向高质量检索动作”。
```

## 9. ORPO 是怎么实现和训练的

### 9.1 ORPO 在项目中的角色

ORPO 是轻量偏好优化对照：

```text
不依赖 reference model，用 odds-ratio 方式拉开 chosen/rejected action。
```

它的价值主要是：

- 训练流程更轻；
- 实现更简单；
- 可以作为 DPO 的低成本对照。

### 9.2 ORPO 代码位置

```text
scripts/train_retrieval_policy_orpo.py
outputs/retrieval_policy_orpo_torch/
```

### 9.3 ORPO 训练参数

主要参数：

```text
epochs = 120
batch_size = 16
lr = 1e-4
margin = 0.02
odds_coef = 0.25
source_weight = 0.5
topic_weight = 0.3
point_weight = 0.2
```

ORPO 核心目标：

\[
L = L_{SFT} + \lambda L_{OR}
\]

其中：

\[
L_{OR} = -\log \sigma(\log\mathrm{odds}(a^+) - \log\mathrm{odds}(a^-))
\]

### 9.4 ORPO 当前效果

当前 ORPO 结果：

| 指标 | Baseline | ORPO | 变化 |
|---|---:|---:|---:|
| Average Reward | 0.5111 | 0.4198 | -0.0913 |
| Source Hit | 0.3056 | 0.2083 | -9.73pp |
| Topic Hit | 0.8426 | 0.8218 | -2.08pp |
| Point Recall | 0.5278 | 0.4537 | -7.41pp |

结论：

```text
ORPO 在当前检索策略任务上效果一般。
原因是当前任务是小样本、单步离散动作选择，ORPO 的优势不明显；
它更适合低成本偏好优化对照，而不是当前上线主策略。
```

答辩时可以这样讲：

> ORPO 没有作为最终主策略，因为它在当前检索任务上的 Source Hit 和 Topic Hit 不如 LinUCB。保留 ORPO 的意义是证明我们做了偏好优化对照，并验证轻量偏好方法在这个任务上的局限。

## 10. LinUCB 是怎么实现和训练的

### 10.1 LinUCB 在项目中的角色

LinUCB 是当前推荐上线的检索策略。

它是 contextual bandit 方法，更适合当前任务：

```text
一个 query
  -> 选择一个 retrieval action
  -> 得到一次 reward
```

### 10.2 LinUCB 代码位置

```text
scripts/train_retrieval_policy_linucb.py
backend/retrieval/policy.py
outputs/retrieval_policy_linucb/
```

### 10.3 LinUCB 训练参数

```text
mode = online
episodes = 420
alpha = 0.12
epsilon = 0.08
l2 = 1.0
source_weight = 0.5
topic_weight = 0.3
point_weight = 0.2
```

LinUCB 分数：

\[
\mathrm{score}(a)
= \theta_a^\top x
+ \alpha \sqrt{x^\top A_a^{-1}x}
\]

含义：

- \(\theta_a^\top x\)：预测该 action 的收益；
- \(\sqrt{x^\top A_a^{-1}x}\)：不确定性；
- \(\alpha\)：探索强度；
- 最终选 score 最高的 action。

### 10.4 LinUCB 当前效果

| 指标 | Baseline | LinUCB | 提升 |
|---|---:|---:|---:|
| Average Reward | 0.5111 | 0.6963 | +0.1852 |
| Source Hit | 0.3056 | 0.6481 | +34.25pp |
| Topic Hit | 0.8426 | 0.9329 | +9.03pp |
| Point Recall | 0.5278 | 0.5370 | +0.92pp |

结论：

```text
LinUCB 在当前检索策略任务上的综合效果最好。
```

## 11. Dueling Double DQN 是怎么实现和训练的

### 11.1 DDQN 在项目中的角色

Dueling Double DQN 是神经网络检索策略对照。

它同样学习：

```text
当前 query 状态下，各 retrieval action 的 Q 值。
```

### 11.2 DDQN 代码位置

```text
scripts/train_retrieval_policy_dueling_ddqn.py
outputs/retrieval_policy_dueling_ddqn/
```

### 11.3 DDQN 训练参数

```text
mode = online
episodes = 520
epochs = 240
batch_size = 32
lr = 5e-4
epsilon_start = 0.9
epsilon_end = 0.06
epsilon_decay = 0.988
source_weight = 0.5
topic_weight = 0.3
point_weight = 0.2
```

Dueling Q 网络拆成：

\[
Q(s,a) = V(s) + A(s,a) - \frac{1}{|\mathcal{A}|}\sum_{a'} A(s,a')
\]

含义：

- \(V(s)\)：当前 query 状态本身的价值；
- \(A(s,a)\)：某个检索 action 相对其他 action 的优势；
- 这样比普通 Q 网络更稳定地比较动作。

### 11.4 DDQN 当前效果

| 指标 | Baseline | Dueling DDQN | 提升 |
|---|---:|---:|---:|
| Average Reward | 0.5111 | 0.6813 | +0.1702 |
| Source Hit | 0.3056 | 0.6343 | +32.87pp |
| Topic Hit | 0.8426 | 0.9236 | +8.10pp |
| Point Recall | 0.5278 | 0.5185 | -0.93pp |

结论：

```text
DDQN 效果接近 LinUCB，说明神经网络策略也适合这个离散动作任务。
但当前样本量下，LinUCB 更简单、更稳定、更可解释。
```

## 12. 各策略综合对比

| 方法 | 主要定位 | 当前效果结论 |
|---|---|---|
| Baseline | 原始 query 检索 | 对照组 |
| PPO | 策略梯度检索策略 | 有提升，但不如 LinUCB/DDQN |
| DPO | 偏好对检索策略 | 有提升，适合说明偏好优化有效 |
| ORPO | 轻量偏好优化对照 | 当前任务效果一般 |
| LinUCB | Contextual Bandit 检索策略 | 当前综合最好，上线主策略 |
| Dueling DDQN | 神经网络 Q-learning 对照 | 效果接近 LinUCB，但复杂度更高 |

核心结论：

```text
当前项目最适合上线的是 LinUCB。
PPO/DPO/ORPO/DDQN 的价值主要体现在实验对照、偏好优化、多步工具决策扩展。
```

## 15. 为什么说 LinUCB 最好，但 PPO/DPO/ORPO 仍然有意义

LinUCB 最适合当前主任务：

```text
单步 query -> action 选择。
```

PPO 更适合未来扩展：

```text
多步 Agent 行为策略：
是否检索 -> 是否读文件 -> 是否继续搜索 -> 是否生成图表 -> 是否回答。
```

DPO 更适合未来扩展：

```text
答案偏好优化：
更重证据、更少幻觉、更有结构、更符合人类偏好。
```

ORPO 更适合未来扩展：

```text
低成本偏好优化对照：
不依赖 reference model，训练流程更轻。
```

所以答辩时应该说：

```text
在当前检索策略任务上，LinUCB 综合效果最好。
但 PPO/DPO/ORPO 不是无效，它们负责的是更高层的 Agent 行为和答案偏好优化。
```

## 16. 可以放 PPT 的总结

### 16.1 简洁版

> 本项目的强化学习主要用于优化 RAG 检索策略。系统把每个用户问题看成一个状态，把不同检索方式看成动作，通过 Source Hit、Topic Hit 和 Point Recall 计算奖励，训练 PPO、DPO、ORPO、LinUCB 和 DDQN 等策略。实验结果表明，LinUCB 在当前单步检索任务上综合效果最好，因此作为上线主策略。

### 16.2 详细版

> 本项目没有直接微调 Qwen，而是在 Qwen 回答前增加了强化学习检索策略层。对于每个 query，系统先提取问题特征，再由策略模型选择 retrieval action，例如偏强化学习、偏轨迹规划、偏论文方法或扩大检索范围。RAG 根据该 action 改写后的 query 搜索本地论文知识库，并用 Source Hit、Topic Hit、Point Recall 计算 reward。PPO/DPO/ORPO/DDQN/LinUCB 都在同一批问题上评估，其中 LinUCB 在 Source Hit 和 Topic Hit 上表现最好，证明强化学习策略能让 Agent 更准确地找到证据。

### 16.3 一句话答辩

> 强化学习在本项目里主要不是让 Qwen 更会说，而是让 Agent 更会找证据。

## 18. 最终口径

最终可以这样说：

```text
本项目的强化学习模块主要负责优化 Agent 的检索决策。
RAG 提供知识库证据，Qwen 负责生成回答，而强化学习策略负责选择最合适的 retrieval action。
训练时，一个用户问题就是一个 episode，策略选择 action 后执行 RAG 检索，再根据 Source Hit、Topic Hit 和 Point Recall 计算 reward。
在 PPO、DPO、ORPO、LinUCB 和 Dueling DDQN 的对照中，LinUCB 在当前单步检索任务上效果最好，因此作为上线主策略；
PPO/DPO/ORPO 则作为多步 Agent 行为优化和答案偏好优化的扩展方向。
```
